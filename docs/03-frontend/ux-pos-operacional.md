# DISATEQ VENDOR™ — UX POS Operacional

## Purpose

DISATEQ VENDOR™ define una UX operacional basada en continuidad runtime, velocidad contextual y ergonomía operacional prolongada.

La UX debe priorizar:
- continuidad operacional
- velocidad de interacción
- estabilidad contextual
- baja fricción cognitiva
- keyboard-first real
- scanner-native real
- orientación espacial consistente
- persistencia runtime

La experiencia NO debe sentirse como:
- dashboard administrativo
- ERP web tradicional
- POS tabletizado
- SPA multipágina
- CRUD operacional

Debe sentirse como:
- runtime operacional vivo
- entorno persistente
- operación continua
- software desktop-native

---

## Operational Runtime

Toda interacción debe sentirse como continuidad del mismo runtime operacional.

La operación NO debe percibirse como:
- navegación multipágina
- cambio de pantalla
- recarga visual
- transición entre sistemas aislados

Toda mutación contextual debe sentirse como:
- expansión operacional
- continuidad runtime
- profundidad contextual
- mutación natural del mismo entorno operacional

---

## Keyboard-First Runtime

DISATEQ VENDOR™ prioriza interacción keyboard-first como núcleo operacional principal.

La UX debe favorecer:
- velocidad operacional
- continuidad de foco
- reducción de clicks
- navegación rápida
- muscle memory operacional

Toda operación frecuente debe poder resolverse:
- sin mouse
- sin overlays innecesarios
- sin navegación profunda
- sin ruptura contextual

La interacción keyboard-first debe sentirse:
- natural
- consistente
- resiliente
- continua

---

## Scanner-Native Operation

El escáner debe comportarse como extensión natural del runtime operacional.

La UX scanner-native debe priorizar:
- foco persistente
- velocidad inmediata
- continuidad contextual
- mínima interrupción visual
- tolerancia operacional

El sistema debe evitar:
- pérdida innecesaria de foco
- overlays invasivos
- interrupciones runtime
- validaciones destructivas
- bloqueo contextual agresivo

Especialmente en:
- ventas
- búsqueda operacional
- cobro
- conciliación rápida
- flujos de alta velocidad

## Touch Compatibility

DISATEQ VENDOR™ debe ser touch-compatible, pero NO touch-first.

La compatibilidad touch debe existir sin degradar:
- densidad operacional
- velocidad keyboard-first
- continuidad runtime
- precisión operacional

El soporte touch debe funcionar como:
- compatibilidad operacional adicional
- tolerancia contextual
- adaptación secundaria runtime

NO como geometría principal del sistema.

---

## Anti-Patterns

DISATEQ VENDOR™ debe evitar:

- navegación multipágina
- overlays invasivos
- exceso de modales
- dashboards SaaS
- UX mobile-first
- fragmentación contextual
- wizardización excesiva
- routing agresivo
- pérdida de foco runtime
- toolbarización excesiva
- CRUD operacional tradicional
- tabletización innecesaria

Toda decisión UX debe evaluarse mediante:

“¿esto preserva continuidad operacional?”

y:

“¿esto ayuda realmente al operador durante sesiones extensas?”
