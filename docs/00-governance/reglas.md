# DISATEQ VENDOR™ — Reglas Globales

## Estado del Documento

Documento autoridad permanente.

Este documento define las reglas estructurales, restricciones globales y disciplina operacional obligatoria de DISATEQ VENDOR™.

Toda decisión:
- UX
- runtime
- arquitectura
- engineering
- automatización
- workflow
- expansión futura

debe respetar estas reglas.

---

# Regla Operacional Madre

Toda decisión debe evaluarse mediante:

```text
“¿esto preserva continuidad operacional?”
```

y:

```text
“¿esto ayuda realmente al operador durante sesiones prolongadas?”
```

---

# Reglas Arquitecturales

## Runtime-First

DISATEQ VENDOR™ debe diseñarse como:

```text
runtime operacional persistente
```

NO como:
- SPA multipágina
- dashboard SaaS
- conjunto de módulos aislados
- CRUD administrativo tradicional

---

## Continuidad Espacial Obligatoria

Toda expansión contextual debe preservar:
- orientación espacial
- estabilidad visual
- geometría compartida
- continuidad runtime
- muscle memory operacional

Evitar:
- reflows agresivos
- ruptura geométrica
- overlays destructivos
- navegación fragmentada

---

## Mutación Contextual Sobre Multipágina

La navegación debe sentirse como:
- expansión contextual
- profundidad operacional
- continuidad runtime

NO como:
- cambio de aplicación
- routing tradicional
- transición multipágina

---

## Workspace Compartido

El runtime debe preservar:
- geometría madre
- boundaries consistentes
- spacing estable
- alineación operacional

Toda nueva superficie debe integrarse al runtime compartido.

---

## SheetWorks Como Modelo Oficial

Las SheetWorks representan:
- superficies operacionales vivas
- continuidad contextual
- persistencia runtime

NO:
- tabs tradicionales
- páginas web
- módulos aislados

---

# Reglas UX/UI

## Keyboard-First Obligatorio

Toda operación frecuente debe poder resolverse:
- rápidamente
- por teclado
- con mínima fricción
- sin navegación profunda

La UX nunca debe depender principalmente del mouse.

---

## Scanner-Native Obligatorio

El scanner debe comportarse como extensión natural del runtime.

Evitar:
- pérdida de foco
- interrupciones agresivas
- overlays invasivos
- recaptura constante de foco

---

## Color = Semántica Operacional

El color NO es decoración.

El color comunica:
- estado operacional
- intención
- prioridad
- riesgo
- contexto

Semántica oficial:
- verde → confirmar / continuar
- rojo → cerrar / irreversible
- ámbar → advertencia / revisión
- azul/navy → navegación / contexto

---

## Densidad Operacional Obligatoria

La interfaz debe preservar:
- velocidad visual
- proximidad contextual
- claridad operacional
- ergonomía prolongada

Evitar:
- whitespace excesivo
- layouts inflados
- cards decorativas
- dashboardización

---

## Modern Operational UI Obligatorio

La UI debe sentirse:
- desktop-native
- operacional
- contextual
- continua
- estable

NO:
- SaaS genérico
- ERP visual tradicional
- admin template
- mobile app gigante

---

# Anti-Patterns Oficiales

DISATEQ VENDOR™ debe evitar:

- dashboards SaaS
- ERP accidental
- mobile-first accidental
- widgetización
- toolbarización excesiva
- routing multipágina
- modalitis
- overlays invasivos
- CRUD operacional tradicional
- wizardización innecesaria
- arquitectura enterprise artificial
- sobreingeniería
- abstracción prematura
- complejidad innecesaria

---

# Reglas de Arquitectura Técnica

## Repo Real Sobre Teoría

Toda decisión debe:
- verificar runtime real
- inspeccionar repo real
- respetar arquitectura existente
- validar implementación real

El repo manda sobre teoría abstracta.

---

## Implementación Progresiva

Toda complejidad debe introducirse:
- progresivamente
- justificadamente
- operacionalmente validada

Evitar:
- infraestructura prematura
- modularización artificial
- abstracción especulativa

---

## Offline-First Obligatorio

La operación nunca debe depender completamente de:
- internet
- cloud
- sincronización activa
- servicios externos

La persistencia local es prioridad operacional.

---

## Escalabilidad Sin Romper Simplicidad

Toda expansión futura debe preservar:
- claridad operacional
- continuidad runtime
- ergonomía operacional
- simplicidad contextual

---

# Reglas Workflow

## Validación Antes de Consolidar

Toda implementación debe:
- ejecutarse
- validarse
- corregirse
- consolidarse

antes de expandirse.

---

## Consolidar Antes de Expandir

Antes de:
- crear nuevos módulos
- expandir arquitectura
- generar documentación nueva
- abstraer componentes

primero:
- consolidar autoridad
- eliminar drift
- estabilizar naming
- validar runtime real

---

## Separación de Capas

Separar claramente:
- governance
- visión
- runtime
- UX
- design-system
- engineering
- workflow

Evitar:
- documentos monolíticos
- duplicación conceptual
- inflación documental

---

## No Inflación Documental

NO crear documentos:
- redundantes
- vacíos
- especulativos
- artificiales
- enterprise-wiki

Todo documento debe justificar:
- autoridad
- utilidad operacional
- continuidad conceptual

---

# Regla Final

La complejidad interna nunca debe trasladarse innecesariamente al operador.

Toda evolución debe sentirse:
- natural
- progresiva
- estable
- operacionalmente simple.
